# Cache Components

## 用途

让这个网站随着内容增长仍保持统一、轻盈、可维护。任何新增界面都应先寻找可复用结构，再决定是否真的需要新组件。

## 何时使用

- 新增页面、区块、卡片、标签、列表、媒体模块
- 准备重做已有 UI，但不确定是否已有相近实现
- 发现不同页面出现相似结构时
- 需要整理组件库、设计 token 或内容模块时

## 行为规则

- 先盘点现有组件、样式类、排版模式、按钮、列表、标签和媒体容器，再提出新增。
- 优先复用已有组件并通过参数、变体、插槽或内容模型扩展。
- 如果必须新增组件，要说明为什么现有组件无法承担该职责。
- 新组件要服务长期内容增长，尤其支持文章、观察、实验、地点、媒体和时间线等内容类型。
- 维护统一的 spacing、边框、阴影、字体层级和交互反馈，避免每页都重新发明一套 UI。
- 优先抽取通用结构，例如：
  - 文章元信息
  - 主题标签
  - 注释块
  - 媒体说明
  - 地点条目
  - 时间线节点
  - 相关内容链接
- 不要为了“组件化”把简单静态内容过度抽象。

## 输出要求

- 先列出现有可复用项，再列出新增项。
- 对新增项说明：
  - 组件职责
  - 与现有组件的差异
  - 未来可复用场景
  - 建议放置位置
- 如果发现重复实现，要指出合并建议。

## 注意事项

- 这个网站不是后台系统，不要把所有内容都做成标准化企业组件。
- 保留个人网站需要的灵活性，但不能以“自由”为名让 UI 失控。
- 不要复制粘贴一个外观相近、逻辑重复的新组件来图省事。

## 适合本项目的调用场景

- “新增随笔卡片前先调用 `cache-components`，看看能不能复用已有内容卡。”
- “请用 `cache-components` 检查文章页、地点页、媒体页是否出现了重复 UI。”
- “我想做一套 Topic 标签和 Note 卡片，先判断哪些可以抽成共用组件。”
